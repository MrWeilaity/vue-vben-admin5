# 字典管理模块设计与落地方案

## 1. 模块定位与边界

### 1.1 适合承载的数据

- **可枚举、低频变更、高频读取**的数据：状态、类型、开关、标签、来源渠道、业务分类等。
- 需要统一前后端语义的枚举：如 `order_status`、`user_gender`、`enable_flag`。
- 需要附带前端展示元数据的数据：颜色、标签类型、排序、默认项、提示文案。

### 1.2 不适合承载的数据

- 事务型主数据：商品、客户、合同等（应建业务主表）。
- 高速变化运行态配置：秒级频繁修改值（应使用配置中心/参数中心）。
- 强关系模型数据：需要复杂关联与生命周期管理的数据。

### 1.3 与“参数配置”“业务主数据”的区别

- **字典**：离散枚举值集合（多条记录、可排序、可启停）。
- **参数配置**：系统键值配置（单值，多用于开关或阈值）。
- **业务主数据**：业务实体（有完整生命周期和业务规则）。

## 2. 整体架构分层

### 2.1 数据库层

- `sys_dict_type`：字典类型元数据。
- `sys_dict_data`：字典项数据。
- 冗余 `type_code` 字段避免多表联查，提高按编码读取性能。

### 2.2 领域模型层

- `SysDictType` / `SysDictData` 实体。
- 支持排序、状态、默认值、标签样式扩展、JSON 扩展字段。

### 2.3 服务层

- `ISysDictService` + `SysDictService` 实现。
- 封装核心流程：类型/字典项增删改查、编码变更联动、默认项唯一、缓存刷写。

### 2.4 缓存层

- 仅使用 Redis 作为统一缓存层（避免多实例本地缓存一致性问题）。
- Key 设计：`dict:data:{typeCode}`。
- 变更触发精确失效，支持全量刷新与启动预热。

### 2.5 接口层

- 后台管理接口：类型与字典项维护（分页、状态、删除、刷新缓存）。
- 业务消费接口：按编码与批量编码读取（供前端 `useDict` 或其他模块复用）。

### 2.6 前端消费层

- 新增 `useDict` 组合式能力：缓存、批量预取、value->label 映射。
- 管理页采用“类型 + 字典项”双表联动，贴合若依类操作习惯。

## 3. 表结构建议

### 3.1 字典类型表 `sys_dict_type`

- `id`：主键。
- `name`：字典名称。
- `code`：唯一业务编码（唯一约束）。
- `status`：启停。
- `cache_enabled`：是否参与缓存。
- `built_in`：内置标识（防误删）。
- `sort_order`：排序。
- `remark`：备注。
- `create_time/update_time`：审计字段。

### 3.2 字典项表 `sys_dict_data`

- `id`：主键。
- `type_id`：关联类型ID（外键）。
- `type_code`：冗余编码。
- `label/value`：显示文本与存储值。
- `status`：启停。
- `is_default`：默认值。
- `sort_order`：排序。
- `tag_type/tag_class/css_style`：前端展示扩展。
- `ext_json`：JSON 扩展数据。
- `remark/create_time/update_time`：备注与审计。
- 唯一约束：`(type_code, value)`。

### 3.3 索引建议

- `idx_sys_dict_type_status_sort(status, sort_order, id)`。
- `idx_sys_dict_data_type_status_sort(type_code, status, sort_order, id)`。
- `idx_sys_dict_data_type_id(type_id)`。

## 4. 核心业务流程（摘要）

1. 新增类型：校验编码唯一 -> 入库。
2. 维护字典项：校验同类型 value 唯一 -> 默认项互斥 -> 入库。
3. 修改类型编码：更新类型表 -> 级联更新字典项冗余编码 -> 失效缓存。
4. 删除类型：校验非内置 & 无字典项引用 -> 删除。
5. 启停：更新状态 -> 失效缓存。
6. 查询：管理端分页 + 业务端按编码读取。
7. 业务表使用：业务表持久化 `value`，前端回显时通过 `useDict` 转 label/tag。

## 5. 缓存一致性策略

- 读路径：先查 Redis，未命中查 DB 后回填 Redis。
- 写路径：DB 成功后删除对应 `dict:data:{typeCode}` key，读时懒加载回填。
- 全量刷新：清理所有 `dict:data:*` key 后按类型全量重建。
- 启动预热：应用启动时将所有启用缓存的字典类型一次性写入 Redis。
- 编码修改时：同时删除旧编码和新编码缓存，避免脏读。

## 6. 前端统一使用方式

- 新增 `useDict(typeCode)`：
  - `load()` 拉取字典；
  - `selectOptions` 直接供 `Select`；
  - `getLabel(value)` 统一回显。
- 新增 `preloadDict(typeCodes)`：复杂页面进入前批量预热字典。
- 管理页面通过 `help` 字段输出问号提示，减少误操作。

## 7. 扩展性设计建议

- 多租户：在类型与字典项加 `tenant_id` + 联合唯一约束。
- 国际化：增加 `label_i18n` JSON 或独立翻译表。
- 分组：增加 `group_code` 便于归类展示。
- 版本控制：新增快照表，支持回滚。
- 引用检查：提供“业务引用统计/拦截删除”机制。
- 导入导出：CSV/Excel 模板，支持覆盖与增量导入。

## 8. 风险与规范

- 编码规范：`^[a-z][a-z0-9_:.]{1,63}$`，建议 `业务域_语义`。
- 删除策略：内置字典禁止删除；已引用字典建议软删或拦截。
- 停用策略：停用优先于删除，避免历史数据断层。
- 兼容策略：业务表存储 `value` 时需保持历史值可识别。

## 9. 推荐实现顺序

1. 数据库迁移脚本（表+索引+菜单权限）。
2. 后端实体/DTO/Service/Controller。
3. Redis + 本地缓存能力。
4. 前端管理页（类型+字典项双表联动）。
5. 前端 `useDict` 消费封装并逐步替换业务硬编码。
6. 最后补充导入导出、引用检查、国际化增强。
